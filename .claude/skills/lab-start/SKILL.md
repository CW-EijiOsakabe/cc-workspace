# lab:start: 技術Lab開始コマンド

技術的な調査やMVP作成などのお題ごとに専用フォルダを作成し、前提情報をヒアリングして steering.md を生成するコマンドなのだ。

---

## User Input

```text
$ARGUMENTS
```

---

## パラメータ解析

User Input から以下を抽出:

- **topic**: お題・テーマ（必須）

### slug化ルール

- 日本語の場合: 内容を表す短い英語のslugに変換（例: "WebSocketでリアルタイムチャット" → `realtime-chat-websocket`）
- 英語の場合: lowercase + ハイフン区切り（例: "Build a CLI tool" → `build-cli-tool`）
- 最大40文字程度に抑える

---

## 実行フロー

```
Phase 1: フォルダ作成 → Phase 2: ヒアリング → Phase 3: steering.md 作成
```

全てメインコンテキストで実行する（Agent呼び出し不要）。

---

## Phase 1: フォルダ作成

### Step 1: slug生成と日付取得

- 今日の日付を `YYYYMMDD` 形式で取得
- topic からslugを生成

### Step 2: フォルダ作成

```bash
mkdir -p lab/sessions/YYYYMMDD-<slug>
```

例: `lab/20260306-realtime-chat-websocket/`

### Step 3: 重複チェック

同じ日付・slugのフォルダが既に存在する場合は、ユーザーに確認してから上書きするか、slug末尾に連番を付与する（例: `-2`）。

→ **Phase 2 へ進む**

---

## Phase 2: ヒアリング（前提情報の収集）

お題について良い形で進めるために、前提情報を `AskUserQuestion` で収集するのだ。

### 質問設計の方針

お題の内容によって聞くべきことは変わるが、以下の観点を参考にして**最も重要な2〜3問**を選んで質問する。全部聞く必要はなく、お題から推測できるものはスキップして良い。

| 観点 | 聞くべきケース |
|------|--------------|
| ゴール・成果物 | お題から成果物が明確でない場合（「〜について調査」のような曖昧な表現） |
| 技術スタック・制約 | 実装系のお題で、使用技術が指定されていない場合 |
| 背景・動機 | なぜこのお題に取り組むのか文脈が不明な場合 |
| スコープ | お題が広すぎて何をどこまでやるか不明な場合 |
| 既知の情報 | 既に調査済みの情報やベースとなる知識がありそうな場合 |

### 質問例

```
AskUserQuestion({
  questions: [
    {
      question: "このLabのゴールは何なのだ？",
      header: "ゴール",
      options: [
        { label: "動くMVPを作る", description: "実際に動作するプロトタイプを作成" },
        { label: "技術調査・検証", description: "技術の実現可能性や特性を調べる" },
        { label: "学習・理解を深める", description: "新しい技術やコンセプトを学ぶ" },
        { label: "比較検討", description: "複数の選択肢を比較して判断材料にする" }
      ],
      multiSelect: false
    },
    {
      question: "技術スタックや制約はあるのだ？",
      header: "技術制約",
      options: [
        { label: "特にない（お任せ）", description: "最適な技術を提案してほしい" },
        { label: "言語/FW指定あり", description: "使いたい技術が決まっている" },
        { label: "既存プロジェクトに組み込み", description: "既存のコードベースに合わせる必要がある" }
      ],
      multiSelect: false
    }
  ]
})
```

上記はあくまで例であり、お題に合わせて質問内容を柔軟に調整すること。ユーザーが「Other」で自由記述した場合もきちんと反映する。

→ **Phase 3 へ進む**

---

## Phase 3: steering.md の作成

ヒアリングで集めた情報をもとに `lab/sessions/YYYYMMDD-<slug>/steering.md` を作成するのだ。

### steering.md テンプレート

```markdown
# <お題のタイトル>

> Lab作成日: YYYY-MM-DD

## 概要

<お題の概要を2〜3文で説明。何をやるのか、なぜやるのかを簡潔に。>

## ゴール

- <具体的なゴール1>
- <具体的なゴール2（あれば）>

## 前提・制約

- <ヒアリングで得た技術制約やスコープ>
- <背景情報や既知の情報>
- <その他の前提条件>

## アプローチ（案）

<ゴールに向けてどう進めるかの初期方針。調査ならどこから調べるか、MVP作成ならどういう構成にするか、など。箇条書きで2〜4項目程度。>

1. <ステップ1>
2. <ステップ2>
3. <ステップ3>

## メモ

<追加で残しておきたい情報があれば。なければ空でOK。>
```

### 記載方針

- ヒアリングの回答内容を**そのまま貼り付けるのではなく**、整理・構造化して書く
- ゴールが曖昧な場合は、ヒアリング結果から推測して具体化する
- アプローチは仮のものでよい。作業を進める中で修正されることが前提

---

## 出力形式

全て完了したら以下を表示:

```
Lab を開始したのだ！

お題: <タイトル>
フォルダ: lab/sessions/YYYYMMDD-<slug>/

steering.md を作成したので、ここから作業を進めていくのだ。

次のステップ:
- steering.md を確認して、方針に問題がなければ作業開始
- 方針を変えたい場合は教えてほしいのだ
```

---

## 注意事項

- `lab/` ディレクトリが存在しない場合は自動で作成する
- steering.md はシンプルに保つ。長大なドキュメントにしない
- ヒアリングは最小限にとどめる（ユーザーを質問攻めにしない）
- お題が十分に明確な場合は、質問を1つに減らすか、省略してもよい
